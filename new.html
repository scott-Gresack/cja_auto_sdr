0 UCP Protocol Architecture

Understanding the layered design that powers flexible, AI-native commerce integrations

Modular, Layered Design

UCP's architecture is intentionally layered and modular to avoid monolithic rigidity. This design allows rapid evolution and independent versioning across different functional areas:

üõçÔ∏è

Base Shopping Service

Defines core commerce primitives (checkout session, line items, totals, status). Foundation: dev.ucp.shopping

‚öôÔ∏è

Capabilities

Self-contained functional modules (Checkout, Order, Catalog, Identity Linking). Each versioned independently for stability.

üîå

Extensions

Optional domain-specific add-ons (loyalty, discounts, subscriptions, fulfillment). Merchants can define custom extensions without committee approval.

Transport Protocols & Bindings

UCP is transport-agnostic by design. Merchants advertise supported protocols, agents negotiate which to use:

üåê

REST

HTTPS+JSON for universal compatibility and traditional plugin integration

ü§ñ

MCP (Model Context Protocol)

Direct tool invocation within LLM context for native AI agent integration

üîê

AP2 (Agent Payment Protocol)

Cryptographic mandates for autonomous AI-driven payment authorization

üîÑ

A2A (Agent-to-Agent)

Direct agent communication without HTTP request/response overhead

üì±

EP (Embedded Protocol)

Web-based interface embedding (iframe/SDK) for hosted checkout flows

Discovery & Capability Negotiation

UCP flips the integration burden by making capabilities discoverable and negotiable in real-time:

1

Merchant Profile Publication

Each merchant publishes a profile (JSON document) at a standard URL: https://merchant.com/.well-known/ucp declaring capabilities, extensions, and transport endpoints.

2

Agent Capability Advertisement

Each agent (platform) publishes its own profile describing which UCP features it can work with.

3

Real-Time Negotiation

When an agent initiates a transaction, it discovers the merchant's profile and includes a reference to its own profile in the request. Both sides negotiate which capabilities and extensions to activate.

Result: Agents only request what merchants support. Merchants expose only what they've built. This eliminates custom integrations and allows new agents to work with any merchant instantly‚Äîno manual configuration needed.
